## 1. TL;DR for agents

- Almond **already ships the core loop** (missed-call/lead capture, online booking, intake-form PDF chart write-back, two-way SMS, ROI dashboard). Tiering is mostly an **additive entitlement wrapper over existing endpoints**, not net-new feature build. The 4 genuinely net-new features are **AI voice receptionist, cancellation/waitlist auto-fill, reputation/Google-review automation, insurance eligibility (270/271)**.
- There is **NO plan/entitlement/billing/usage/tenant layer** anywhere today, and even role auth (`auth.require_role`) is a **broken, never-wired stub**. `require_feature` must be built **fresh** on `auth.current_user`, not "paralleled" from `require_role`.
- There is **no Account/Org/Location entity** — only a frozen `User(id,name,email,role,status)`. A new per-location **Account (tenant)** model must come first.
- **Recommended structure:** 3 paid tiers — **Core ~$199 / Pro ~$349 / Practice+ quote-gated $599+**, all **per LOCATION/mo** — plus a **metered add-on layer** (AI voice minutes, SMS-segment overage, payment volume %). Value metric = **per-location** (dental norm; what Open Dental uses). **Never per-seat** (front desks share logins).
- **Do NOT fence the core loop** — it's the price+depth+ease wedge against NexHealth (~$350, churns on price). Price Core decisively under $350. Gate ONLY scale/automation/AI/governance differentiators.
- **Enforcement:** a `require_feature(feature_key)` FastAPI dependency + a static `PLAN_FEATURES` matrix + a `require_quota(meter_key)` dependency. **Our DB is the gating source of truth**; mirror Stripe as billing system of record via webhooks. **Never read Stripe synchronously on the request path.**
- **Additive only:** don't touch frozen `api.py` beyond router includes; don't change frozen `/v1` shapes; gate only new/premium endpoints; existing routes stay ungated per `CONTRACT.md`.
- **Three meters per location/period:** `sms_segments` (campaign/bulk only — exclude transactional/two-way), `ai_minutes`, `eligibility_checks`; payments as Stripe % passthrough. `notify.send_sms()` is the SMS instrumentation point.
- **HIPAA/BAA:** AI voice, payments, and reviews/PII each need signed BAAs before real PHI; ship them in `recorded`/sandbox mode until BAAs land and gate them to Practice+. **Stripe will not sign a BAA — keep PHI out of Stripe entirely.**
- **Migration:** SQLite POC → **Postgres required before metered billing goes live** (SQLite's single-writer lock can't do concurrent usage increments).

## 2. Answers to the defined research questions

### Q1 — Which features to build next, ranked by ROI + WTP, and which tier?
**Answer:** **Core/entry (the wedge — never fence):** (1) missed-call text-back + lead capture *[shipped; ~38% of dental calls go unanswered, text-back recovers 25–60%]*, (2) online self-scheduling *[shipped; 89% demand vs 63% supply]*, (3) two-way SMS inbox *[shipped; table-stakes]*, (4) automated reminders + recall *[22.95% no-show reduction]*. **Pro/mid (differentiators, the upsell):** (5) cancellation/waitlist auto-fill *[NET-NEW; recovers $30–133k/yr; simultaneous-notify fills 75–95% vs 25–35% sequential]*, (6) reputation/Google-review automation *[NET-NEW; +1 star = +5–9% revenue]*, (7) deeper analytics/attribution *[basic shipped; gate cohort/multi-location]*, (8) patient self-rescheduling *[bundle into scheduling]*. **Premium/metered add-on (sliding-scale ceiling, BAA-gated):** (9) AI voice receptionist *[NET-NEW; biggest build + PHI risk; meter by minutes]*, (10) text-to-pay/payments *[monetize on volume %]*, (11) insurance eligibility 270/271 *[NET-NEW dental-depth differentiator; needs clearinghouse connector]*.
**Confidence:** High on ranking/ROI; Medium on exact tier placement (product judgment).

### Q2 — How do competitors package and gate features, at what prices?
**Answer:** The category is **quote-gated** (NexHealth, Podium, Solutionreach, RevenueWell, Tebra, Dental Intelligence, Adit all hide tier prices; only Weave publishes "from $249"). Anchors: **NexHealth ~$350** (churns on price); **Podium ~$399 Core / ~$599 Pro** (lands $500–800 with add-ons); **Weave ~$249 Pro / mid-$300s Elite + $750 setup**; **RevenueWell from ~$189**. Common (not universal) ladder: entry = SMS + reminders + missed-call-text + review requests; mid = scheduling + forms + payments + recall; top = AI + analytics + high bulk-SMS. **Per-location pricing dominates** the engagement loop (only EHR/analytics plays use per-seat). **Transparent published pricing is itself a wedge — do NOT copy quote-gating** (except the top DSO tier).
**Confidence:** High on prices/per-location/quote-gating; Medium on the exact feature ladder (verification showed it's directional — RevenueWell puts scheduling in base, Podium ships basic AI at entry, Adit bundles AI free).

### Q3 — Tier structure + value metric + how many tiers?
**Answer:** **Good-Better-Best = 3 paid tiers (Core / Pro / Practice+)** + a metered usage add-on layer, with Practice+ the quote-gated DSO/multi-location/governance tier (effectively a 4th option). **Value metric = per-location** (per-seat is "a terrible value metric" — shared front-desk logins). Stripe subscription `quantity = location_count`; seats become a soft per-tier cap, not the meter. Metered allotments (AI minutes, SMS segments) per tier + overage credit packs = the sliding-scale lever.
**Confidence:** High.

### Q4 — Entitlements architecture (build vs buy)?
**Answer:** **BUILD in-house** — do not buy. Matrix is tiny (3 plans × ~12 features = a static dict). Components: a stable `feature_key` set; `PLAN_FEATURES: dict[plan -> set[feature_key]]` in a new `entitlements.py`; `require_feature(feature_key)` dependency on `auth.current_user` returning 402 + upgrade payload; `require_quota(meter_key)` for metered features; an `Account` (tenant) model with `plan` + `location_count`; `usage_events`/`usage_counters` tables. Frontend extends `App.tsx` nav gating with a plan check beside `isOwner()`. Do NOT adopt OpenFGA/LaunchDarkly/Stigg for a 3×12 matrix (overkill + a PHI-adjacent third party). **`auth.require_role` is a broken stub — write `require_feature` fresh.**
**Confidence:** High.

### Q5 — Stripe Billing for tiered + metered, with HIPAA caveats?
**Answer:** One Subscription per Account: a **licensed (per-location quantity) Price** for the base tier + **metered Prices** (Billing Meter Events API) for AI minutes, SMS overage, eligibility checks; payments via Stripe Connect/% passthrough. Drive entitlement state from **webhooks** (`customer.subscription.created/updated/deleted`, invoice events) into our DB; **gate from our DB, never call Stripe on the request path**. Report metered usage by pushing aggregated period counts. **Stripe is NOT a HIPAA BAA vendor and forbids PHI** — send only de-identified billing data (account id, location count, meter qty, plan); never patient data in metadata/descriptions. Use a sales-assisted **paid pilot**, not unattended free trial (Almond is implementation-heavy).
**Confidence:** High on architecture; Medium on exact Stripe meter API surface (verify SDK at build time — legacy usage-records API is removed; use Billing Meters).

### Q6 — Exact, minimal-churn codebase extension points?
**Answer (file-by-file):** `models.py` — add `Account`/`Subscription`/`UsageEvent` (don't alter frozen `User`; attach `account_id` via column). `db.py` — add `accounts`/`subscriptions`/`usage_events`/`usage_counters` to `_SCHEMA`; reuse `store._ensure_submission_created_at` additive-migration idiom for `users.account_id`. **NEW `entitlements.py`** — Plan enum, `PLAN_FEATURES`, `require_feature()`/`require_quota()` (on `current_user`; fix, don't copy, the `require_role` bug). `config.py` — add `ALMOND_STRIPE_*` under the existing prefix. `notify.send_sms()` — increment `sms_segments`. **NEW `routers/billing.py` + `routers/usage.py`**, included in `api.py` as additional router includes only. Wrap **new/premium** endpoints with `require_feature`; leave existing `/v1` routes ungated per `CONTRACT.md`. `auth.login` — add `account.plan` + `entitlements` additively. Frontend: `types.ts` (additive Account/plan/entitlements), `api.ts` (billing/usage calls), `App.tsx` (plan check beside `isOwner`, upsell CTAs), `ui.tsx` (`UpgradePrompt`).
**Confidence:** High — verified against current source + frozen contract.

### Q7 — Telemetry to meter, trigger upgrades, and show ROI?
**Answer:** Three per-location/period counters: `sms_segments_used` (campaign/bulk only; exclude transactional/two-way), `ai_minutes_used`, `eligibility_checks_used`; plus payment volume (% passthrough, no cap). Append-only `usage_events(account_id, location_id, meter_key, qty, ts)` incremented **at the point of action** (`notify.send_sms` for SMS; AI handler for minutes; eligibility connector for checks), aggregated into `usage_counters`. Upgrade trigger: surface the add-credits/upgrade CTA **at the moment a cap is hit** (highest-converting). For ROI: extend `reports.py` KPIs with no-show recovered $ (it already models `_FRONT_DESK_HOURLY=26`, `_MINUTES_PER_CALL=11`). **`Reports.tsx` hardcodes the revenue breakdown (~line 121) and ignores the backend's computed `revenue_breakdown`** — wire it to real data as part of the analytics upsell.
**Confidence:** High — counters, instrumentation points, and the Reports.tsx mock all verified.

### Q8 — HIPAA/BAA/SOC2 implications + what gates behind enterprise?
**Answer:** PHI-touching premium features need signed BAAs before real data: **AI voice** (telephony + STT/TTS + LLM subprocessors handle call audio + identity — biggest exposure), **text-to-pay** (Stripe is not a BAA vendor — keep PHI out of payment metadata; use a payments BAA vendor like Rectangle Health/InstaMed for PHI-linked pay), **review automation** (outbound to Google can leak PII; consent-gate + TCPA written consent; confirming patient status in a public reply is a PHI disclosure). **Multi-tenant isolation:** every entitlement/usage/billing query MUST be scoped by `account_id`. **Practical gate:** ship AI voice/payments/reviews/eligibility in `recorded`/sandbox mode (`notify` already returns `status:'recorded'` with no creds) and behind **Practice+** with BAA + SOC2 + audit logging (`audit_log` table exists). Eligibility (270/271) needs a clearinghouse BAA.
**Confidence:** High on requirements; Medium on which specific vendor each feature uses.

### Q9 — Where does subscription/entitlement state live; SQLite→Postgres?
**Answer:** **Our DB is the source of truth for entitlement/gating reads** (plan, feature set, usage counters) — fast, no external dependency on the request path; **Stripe is the billing system of record** and pushes state into our DB via webhooks. Never gate by calling Stripe synchronously. Migration: (1) metered usage needs atomic concurrent increments → **Postgres required before metered billing**; (2) keep new DDL additive + Postgres-portable; (3) reuse the additive-migration idiom for POC SQLite column adds, but adopt **Alembic** for Postgres; (4) replace direct `sqlite3` with a connection abstraction; `usage_counters` use `INSERT ... ON CONFLICT DO UPDATE`.
**Confidence:** High.

## 3. Recommended approach

**Recommendation:** Build a **new, additive entitlement + per-location billing layer that wraps Almond's existing endpoints**; do NOT rebuild the core loop and do NOT fence it. Ship **3 per-location tiers — Core (~$199, full core loop, deliberately under NexHealth's ~$350), Pro (~$349, adds waitlist auto-fill + review automation + advanced analytics + self-rescheduling), Practice+ (quote-gated $599+, multi-location/DSO + governance: SSO, audit, BAA/SOC2, includes AI voice + eligibility + payments)** — plus a **metered usage add-on layer** (AI voice minutes, SMS-segment overage, payment volume %) as the sliding-scale ceiling. Add an `Account` model, a static `PLAN_FEATURES` matrix in `entitlements.py`, a `require_feature()` dependency built **fresh** on `auth.current_user`, a `require_quota()` dependency, three per-location meters, and Stripe Billing with **our DB as gating source of truth and Stripe as billing system of record** (synced via webhooks). Keep PHI out of Stripe. Gate AI voice/payments/reviews/eligibility behind signed BAAs (ship in `recorded`/sandbox until BAAs land) and behind Practice+. Migrate to Postgres before metered billing. Respect frozen `api.py` and `/v1` shapes; emit NO code comments.

**The 3–5 facts that drive it:** (1) the core loop already exists, so this is packaging not building [S8,S9,S10]; (2) no entitlement/tenant layer exists and `require_role` is broken [S1,S4,S5,S15]; (3) per-location is the dental norm; per-seat is a poor value metric for shared logins [S22,S23]; (4) the wedge is being cheaper/easier than NexHealth, so the core loop must stay unfenced and Core priced under $350 [S26 + provided context]; (5) Stripe isn't a BAA vendor, so PHI features need separate BAAs and sandbox-until-signed [S? stripe SSA].

**Alternatives considered and rejected:**

| Option | Why not | When it would win |
|---|---|---|
| Reuse/extend `auth.require_role` for gating | It's a broken stub (no `Depends(current_user)`, zero call sites) — copying propagates an always-401 bug | Only after first *fixing* `require_role`; even then write `require_feature` fresh |
| Per-seat pricing | Front desks share logins → seats decouple from value; caps revenue in small practices | Only if Almond pivots to a per-clinician EHR/analytics product (out of scope) |
| Buy entitlement/flag platform (Stigg/LaunchDarkly/OpenFGA) | 3×12 matrix is a static dict; adds latency + a PHI-adjacent third party | If the matrix explodes into hundreds of per-customer entitlements |
| Stripe as gating source of truth (query per request) | Puts Stripe latency/availability on the request path; outages would gate core features | Never on the request path; OK for admin/reconciliation jobs |
| Quote-gate all pricing (like incumbents) | Transparent per-location pricing is the ease+price wedge; quote-gating is what they churn on | Only the top DSO/Practice+ tier |
| Stay on SQLite for production metering | Single-writer lock → unreliable concurrent increments → billing corruption | Fine for POC + static matrix + low-write rows; not metered counters |
| Unattended self-serve free trial | Implementation-heavy product converts worse self-serve than sales-assisted paid pilot | If zero-touch self-onboarding is ever achieved |

## 4. Findings (by thread)

### 4.1 Feature ranking, ROI, WTP, tiers
- Almond already ships 6 of the core-loop pieces (capture, booking, intake→PDF write-back, two-way SMS, ROI dashboard) — *[High; `routers/core.py`, `messages.py`, `reports.py`, `pdf.py`]*. **Implication:** roadmap is packaging, not build.
- Missed-call text-back is #1 core: ~38% of inbound dental calls unanswered; text-back recovers 25–60% — *[High; peerlogic.com, netpartners.marketing]*.
- Online self-scheduling #2: 89% demand vs 63% supply; only 11% of groups have majority self-scheduling — *[High; MGMA 2024, Experian 2024]*.
- Reminders + recall: **22.95% no-show reduction** across a 1.6M-appointment study — *[High; Sesame study, Dental Tribune 2013; Angle Orthodontist 2018 PMC8288327]*.
- No-show economics: each miss ~$200–400 lost production (crown prep $600–1,200) — *[Med; scottleune.com, curogram.com]*.
- Waitlist auto-fill recovers ~$30–133k/yr; simultaneous notify fills 75–95% vs 25–35% sequential — *[Med; dentistryiq.com, certifyhealth.com]*.
- Reviews: +1 star = +5–9% revenue — *[High; HBS WP 12-016 (Luca 2011)]*.
- AI voice = highest WTP, sold as a separate paid add-on/top-tier by every competitor — *[High; dentalbase.ai, emitrr, wiserreview]*.
- WTP anchors: NexHealth ~$350 (churns), Podium $399/$599, Weave $249+ — *[High; wiserreview 2026-05, getweave]*.

### 4.2 Competitor packaging & gating
- Whole category is **quote-gated**; only Weave publishes a floor — *[High; vendor pricing pages]*.
- **Per-location dominates** the engagement loop; per-seat only for EHR/analytics (Tebra per-provider, Dental Intelligence per-user) — *[High]*.
- NexHealth meter is the cleanest blueprint: **5,000 campaign SMS/loc/mo, $0.012 overage; 800 eligibility checks/mo, $0.20 overage; payments 2.9%+$0.30; no rollover; transactional SMS excluded** — *[High; help.nexhealth.com/articles/11584920, verified 2025-12-10]*.
- The "universal ladder" / "AI always top-tier" / "scheduling is the entry→mid paywall" claims were **REFUTED as only directional** — RevenueWell puts scheduling in base, Podium ships basic AI at entry, Adit bundles AI free. **Treat the ladder as a heuristic, validate fences against your own WTP data.**
- A2P 10DLC carries ~$5/mo/location (Twilio); MMS ≈ 3× SMS cost — *[Med; wiserreview]*.

### 4.3 Tiered-SaaS packaging & value metric
- Good-Better-Best (3 paid tiers) is the de-facto SaaS starting point — *[High (the "72%" stat was downgraded to "~two-thirds"); growthunhinged, aakashg]*.
- Value metric must track perceived value, be predictable, and grow with the customer — *[High; ProfitWell/Campbell]*. Per-seat has a "structural AI problem" once AI does the work — *[High; digitalapplied 2026]*.
- Don't feature-wall the core loop; reserve add-ons for optional value (AI, payments, priority support) — *[High; getmonetizely, growthunhinged]*.
- Sales-assisted paid pilot > self-serve free trial for implementation-heavy products *[High but exact % weakly sourced]*; usage-cap-hit is the highest-converting upgrade trigger — *[High]*.

### 4.4 Entitlements architecture
- Backend is single-tenant: `db.py users` lacks `account_id`; add an `accounts` table + FK first — *[High; `db.py`]*.
- Model = boolean flags + numeric quotas/seats + metered limits via a plan→feature matrix + per-account overrides — *[High; getlago.com]*.
- Don't make a flag vendor the entitlement engine; sync FROM billing — *[High; launchdarkly.com]*.
- Quota race fix: atomic `UPDATE used=used+n WHERE used+n<=limit RETURNING used` (rowcount 0 = deny); fail-closed on PHI — *[High]*.

### 4.5 Stripe Billing (2026)
- **Entitlements API:** `POST /v1/entitlements/features` (lookup_key); `GET /v1/entitlements/active_entitlements?customer=...`; sync via `entitlements.active_entitlement_summary.updated` webhook — *[High; docs.stripe.com/billing/entitlements, fetched 2026-06]*.
- **Metered billing MUST use Billing Meter + Meter Events API**; the legacy `usage_records` API is **REMOVED** — *[High; Stripe migration guide]*.
- `POST /v1/billing/meters` (event_name, aggregation sum/count, customer_mapping by_id); usage via `POST /v1/billing/meter_events` (event_name, payload, optional identifier idempotency key) — *[High]*.
- Checkout `mode=subscription`; Customer Portal for self-serve; mid-cycle changes via `Subscription.update` with proration — *[High]*.
- **Stripe will NOT sign a BAA and forbids PHI** — *[High; stripe.com/legal/ssa modified 2025-11-25; hipaajournal]*.
- Webhook signatures via `stripe.Webhook.construct_event(raw_body, sig, secret)`; SDK `stripe` v15.x (Python 3.9+) — *[Med-High; pypi/stripe]*.

### 4.8 Compliance tiering
- No compliance baseline today: plaintext passwords, no `tenant_id`, hardcoded PMS keys, an unauthenticated PHI-write form endpoint, unused `audit_log` — *[High; backend]*. **(These are pre-real-PHI hardening items.)**
- Top tier (each needs a BAA Almond lacks): AI voice, eligibility 270/271, PHI-linked text-to-pay (Stripe won't sign — use Rectangle Health/InstaMed) — *[High; web]*.
- Review automation = HIPAA marketing + TCPA written consent; public reply confirming patient status is a PHI disclosure — *[High; web]*.
- Lower-mid reuse Twilio + PMS BAAs: reminders, SMS inbox, self-scheduling, waitlist, analytics, conduit pay — *[High; web]*.

### 4.9 Billing persistence / migration
- Persistence is a single-tenant POC: stdlib `sqlite3`, one hardcoded `almond.db`, no pool, every store func opens `db.connect()` directly — *[High; `db.py`, `store.py`, `config.py`, `registry.py`]*.
- Recommendation: **our Postgres authoritative for runtime entitlement checks (local mirror); Stripe authoritative for billing**, synced by webhook — *[High; docs.stripe.com/billing/entitlements, webhooks]*.

## 6. Constraints, gotchas & failure modes

- **`auth.require_role` is a dead, broken stub** — copying it for `require_feature` inherits an always-401 bug. Build fresh with `user: User = Depends(current_user)`.
- **`api.py` and `/v1` shapes are FROZEN.** Adding plan fields to existing bodies (e.g. `User`) breaks the contract. Add fields only to the login envelope; gate only NEW endpoints.
- **No tenant/account/location entity.** Every entitlement/usage/billing query MUST be scoped by `account_id` or you create cross-tenant PHI/billing leaks. `connector_for('demo')` must be parameterized per account.
- **SQLite single-writer lock** makes concurrent `usage_event` increments unreliable. Do NOT enable metered billing on SQLite — move `usage_counters` to Postgres with `INSERT ... ON CONFLICT DO UPDATE` first.
- **Stripe is NOT a BAA vendor.** Never put PHI in Stripe metadata/descriptions/line-item names — only de-identified billing data.
- **Never gate by calling Stripe synchronously** — sync via webhooks into our DB and gate from the DB.
- **Do NOT fence the core loop** (missed-call text-back, SMS inbox, self-scheduling, reminders, chart write-back). Fencing kills the wedge. Gate ONLY differentiators/AI/governance.
- **Chart write-back is NOT a differentiator** — NexHealth has it. It's table-stakes in Core.
- **`Reports.tsx` hardcodes the revenue breakdown** and ignores the backend `revenue_breakdown`; KPI deltas are illustrative. The analytics upsell must replace these mocks or you're selling a fake dashboard.
- **HOUSE RULE: emit absolutely NO code comments** (no inline/block/JSDoc/docstring-as-comment). Self-documenting names only.
- **Meter only campaign/bulk SMS** — never count transactional/two-way SMS against the cap (NexHealth pattern) or you punish normal communication and invite disputes.
- The competitor "universal ladder"/"AI always top-tier"/"scheduling is the entry→mid paywall" claims were **REFUTED as directional only** — validate fences against your own WTP data.
- **MMS ≈ 3× SMS** (Podium meters 3 credits/MMS); **A2P 10DLC** ~$5/mo/location via Twilio — budget/absorb/pass-through.
- **Insurance eligibility (270/271)** needs a clearinghouse/payer connector beyond the PMS connector layer — real new integration, BAA-gated, later-roadmap.
- Migrating off direct `sqlite3` (`db.connect`) needs a **connection abstraction** first — routers/store call `db.connect()` directly, so a blanket Postgres swap touches many call sites.

## 8. Glossary / entities

- **System of engagement vs system of record** — Almond is the engagement layer (booking/forms/messaging) on top of the PMS/EHR (the record).
- **NexHealth** — the dominant incumbent doing Almond's loop across ~70 PMS; ~$350/mo, churns on price. The price/ease wedge target.
- **Entitlement** — what a plan is allowed to do; here a `feature_key` ∈ `PLAN_FEATURES[plan]` plus numeric quotas.
- **`require_feature` / `require_quota`** — the FastAPI dependencies (to build) that enforce plan gating and usage caps.
- **Value metric** — the unit you charge by; here **per location** (not per seat).
- **270/271** — the X12 EDI insurance eligibility request/response pair; needs a clearinghouse connector.
- **A2P 10DLC** — carrier registration for application-to-person SMS; ~$5/mo/location via Twilio.
- **BAA** — HIPAA Business Associate Agreement; required from any vendor (subprocessor) that touches PHI. **Stripe won't sign one.**
- **Stripe Billing Meters / Meter Events** — the current (2026) usage-based billing API; the legacy `usage_records` API is removed.
- **Open Dental** — Almond's live PMS connector; prices per location (the dental norm informing the value metric).
